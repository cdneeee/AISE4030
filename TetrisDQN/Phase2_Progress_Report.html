3. Comparison with Phase 1 Proposal

Observation Space

The confirmed observation space (Box, float32, shape (220,)) is consistent with the Phase 1 proposal. Phase 1 defined the state as st = (B, pt, ot, xt, yt, pt+1), where B is the binary 20×10 board, pt is the current piece type, ot is its orientation, (xt, yt) is its bounding-box position, and pt+1 is the next-piece preview. Our implementation encodes this as 200 board values, plus 7-dimensional one-hot vectors for each piece, a 4-dimensional one-hot for orientation, and two normalised scalars for position — totalling 220 features, all in [0, 1]. This directly reflects the proposed formulation.

Action Space

The confirmed action space (Discrete(6)) matches the Phase 1 proposal exactly. Phase 1 listed: move left, move right, rotate clockwise, rotate counter-clockwise, drop downwards, and no action — all six are present in the implementation.

Potential Challenges

  • Sparse rewards early in training: Line clears are rare for an untrained agent, so the agent will primarily experience the −0.01 step penalty and occasional board-quality penalties. Epsilon-greedy exploration and the high discount factor (γ = 0.99) should help the agent eventually discover line-clearing behaviour.
  • Training speed: A single episode can run thousands of steps. With 5 000 training episodes and a CPU-only device, total training time may be substantial. We plan to profile early runs and may reduce episode count or add hard-drop bias during exploration if needed.
  • Reward shaping balance: The relative weights for hole and height penalties (−0.5 and −0.3 respectively) were chosen heuristically. Tuning these in Phase 3 may be required if the agent learns to minimise penalties at the expense of scoring.

4. Implementation Approach

We are implementing the DQN algorithm from scratch using PyTorch, without relying on a high-level RL library such as Stable Baselines3 or RLlib. This decision is justified by two reasons:

  1. Full control for comparison: Our Phase 3 advanced algorithm is a Dueling DQN, which differs from the baseline only in network architecture (splitting the final layers into a Value stream V(s) and an Advantage stream A(s,a)). Implementing both from scratch ensures that any performance difference is attributable solely to the architectural change, not to library-level differences in replay sampling, target updates, or optimisers.
  2. Custom environment: Since we built a custom Tetris environment rather than using a pre-packaged one, the environment itself is already hand-crafted. Implementing the agent from scratch maintains consistency and gives us complete visibility into every component of the system.

The DQN baseline uses a Multi-Layer Perceptron (512 → 256 → 128 → 6) with an experience replay buffer (capacity 100 000), a target network synced every 1 000 gradient steps, and linear epsilon decay from 1.0 to 0.05 over 100 000 steps. All hyperparameters are centralised in config.yaml.
